# AeroSpace Architecture

This document is a maintainer-oriented map of AeroSpace internals. It is meant to help future changes land in the right subsystem, preserve important invariants, and avoid accidental regressions in focus, layout, workspace assignment, command parsing, or Accessibility integration.

## 1. Repository and build targets

AeroSpace is organized as a Swift package with a thin app wrapper and a separate CLI client.

| Path | Purpose |
| --- | --- |
| `Package.swift` | SwiftPM manifest. Defines `Common`, `AppBundle`, `AeroSpaceApp`, `Cli`, `PrivateApi`, and tests. |
| `Sources/Common/` | Shared client/server types: command args parsing, command help constants, client/server protocol models, basic utilities. |
| `Sources/AppBundle/` | The actual window manager server/runtime. Holds config, commands, tree model, layout, focus, AX integration, UI, observers, socket server. |
| `Sources/AeroSpaceApp/` | Minimal SwiftUI app entry point that calls `initAppBundle()`. |
| `Sources/Cli/` | `aerospace` CLI client. Parses args locally, sends request to the app server over a Unix socket, prints result. |
| `Sources/PrivateApi/` | C bridge for private macOS APIs, such as `_AXUIElementGetWindow` and SkyLight focus helpers. |
| `Sources/AppBundleTests/` | Unit tests for config, command parsing/execution, tree behavior, shell, etc. Uses test app/window stubs. |
| `docs/` | Asciidoc user docs and command docs. Command synopsis blocks generate Swift help constants. |
| `dev-docs/` | Maintainer-facing docs, including this file. |
| `generate.sh`, `script/` | Code/doc generation scripts, release/build helpers. |

Important generated files:

- `Sources/Common/cmdHelpGenerated.swift` is generated from `docs/aerospace-*.adoc` synopsis blocks.
- `Sources/Cli/subcommandDescriptionsGenerated.swift` is generated by `generate.sh`.
- `Sources/Common/gitHashGenerated.swift` and `Sources/Common/versionGenerated.swift` are generated metadata.

## 2. Runtime process model

AeroSpace has two main executables:

1. `AeroSpace.app`: long-running server/window manager.
2. `aerospace`: short-lived CLI client.

The app is authoritative for window management. The CLI only parses enough to report local syntax/help errors and then sends the request to the app server.

### CLI/server command flow

```text
user shell
  |
  v
aerospace CLI
  - parse args with Common cmd parser
  - show local help/errors if needed
  - send ClientRequest to app server
  |
  v
AeroSpace.app Unix socket server
  - parse request again
  - run command inside runLightSession
  - return stdout/stderr/exit code
  |
  v
CLI prints output and exits
```

Key files:

- CLI entry: `Sources/Cli/_main.swift`
- Server: `Sources/AppBundle/server.swift`
- Client/server protocol models: `Sources/Common/model/clientServer.swift`
- Command args: `Sources/Common/cmdArgs/`
- Command implementations: `Sources/AppBundle/command/`

## 3. Startup lifecycle

App startup begins in `Sources/AeroSpaceApp/AeroSpaceApp.swift`, which calls `initAppBundle()` in `Sources/AppBundle/initAppBundle.swift`.

High-level order:

1. Initialize termination handling and parse app/server args.
2. Wait for Accessibility permission.
3. Load default config, then user config.
4. Apply one-shot workspace-to-monitor assignments on connect/startup.
5. Start Unix socket server.
6. Register global observers.
7. Initialize workspace registry and focus an initial workspace.
8. Run a non-cancellable startup heavy refresh.
9. Run startup light session for smart layout and `after-startup-command`.

Sketch:

```text
AeroSpaceApp.init
  -> initAppBundle
     -> waitForAccessibilityPermission_nonCancellable
     -> bootstrapConfig_nonCancellable
     -> reloadConfig_nonCancellable
     -> applyWorkspaceToMonitorAssignmentsOnConnect
     -> startUnixSocketServer
     -> GlobalObserver.initObserver
     -> Workspace.garbageCollectUnusedWorkspaces
     -> runHeavyCompleteRefreshSession(.startup)
     -> runLightSession(.startup) { smartLayoutAtStartup; afterStartupCommand }
```

The startup heavy refresh is intentionally non-cancellable so `isStartup` semantics propagate consistently.

## 4. Configuration architecture

Config model and parser live in:

- `Sources/AppBundle/config/Config.swift`
- `Sources/AppBundle/config/parseConfig.swift`
- specialized parsers in `Sources/AppBundle/config/parse*.swift`

`Config` is a mutable value type stored globally as:

```swift
@MainActor var config: Config = defaultConfig
```

Config loading flow:

```text
readConfig(forceConfigUrl:)
  -> findCustomConfigUrl / defaultConfigUrl
  -> TOMLDecoder -> OrderedJson
  -> parseConfig
     -> top-level configParser registry
     -> specialized parser functions
     -> ConfigParserContext(errors, warnings)
  -> reloadConfig_nonCancellable updates global config if valid
```

Config discovery order is in `ConfigFile.swift`:

1. `--config-path` when provided.
2. `~/.aerospace.toml`.
3. `$XDG_CONFIG_HOME/aerospace/aerospace.toml`.
4. Bundled/default config.

### Adding a config option

1. Add field/default in `Config`.
2. Add parser function or reuse an existing parser.
3. Register it in `configParser` in `parseConfig.swift`.
4. Add tests in `Sources/AppBundleTests/config/ConfigTest.swift`.
5. Update `docs/config-examples/default-config.toml` and relevant docs.
6. If exposed via `aerospace config`, update config-command map/tests.

### Workspace-to-monitor assignment configs

There are two distinct workspace monitor assignment concepts:

#### Permanent force assignment

```toml
[workspace-to-monitor-force-assignment]
    1 = 'main'
    2 = 'secondary'
```

This is hard/continuous. `Workspace.workspaceMonitor` always prefers `forceAssignedMonitor`, and `move-workspace-to-monitor` is blocked for force-assigned workspaces.

#### One-shot assignment on connect/startup

```toml
[workspace-to-monitor-assignment-on-connect]
    1 = { if = 'secondary', larger-than = 'main' }
    2 = 'secondary'
```

This sets the workspace's default `assignedMonitorPoint` during startup/monitor rearrangement, but it does not force future placement. `move-workspace-to-monitor` can change the workspace afterwards.

The resolver lives in `MonitorDescriptionEx.swift` and supports monitor patterns (`main`, `secondary`, number, regex) and optional size predicates (`larger-than`, `smaller-than`). Size comparison uses monitor area (`Rect.area`).

## 5. Command architecture

Commands have two halves:

1. Shared argument parsing in `Sources/Common/cmdArgs/`.
2. Server-side execution in `Sources/AppBundle/command/`.

### Command parsing flow

```text
raw args/string
  -> parseCmdArgs / parseCommand
  -> typed CmdArgs
  -> CmdArgs.toCommand()
  -> concrete Command.run(env, io)
```

Important files:

| File | Role |
| --- | --- |
| `Sources/Common/cmdArgs/parseCmdArgs.swift` | `CmdArgs`, `CmdParser`, parser metadata. |
| `Sources/Common/cmdArgs/parseSpecificCmdArgs.swift` | Core flag/positional parser loop. |
| `Sources/Common/cmdArgs/SubArgParser.swift` | Reusable flag parsers like `--window-id`, `--workspace`. |
| `Sources/Common/cmdArgs/cmdArgsManifest.swift` | `CmdKind` and subcommand parser registry. |
| `Sources/Common/cmdArgs/impl/*CmdArgs.swift` | Per-command typed args. |
| `Sources/AppBundle/command/cmdManifest.swift` | Maps typed args to concrete commands. |
| `Sources/AppBundle/command/impl/*Command.swift` | Command runtime implementation. |
| `Sources/AppBundle/shell/Shell.swift` | Shell command sequences and command execution wrapper. |

`Command` protocol:

```swift
protocol Command: AeroAny, Equatable, Sendable {
    associatedtype T: CmdArgs
    var args: T { get }
    @MainActor func run(_ env: CmdEnv, _ io: CmdIo) async -> T.ExitCodeType
    var shouldResetClosedWindowsCache: Bool { get }
}
```

`Shell.run()` is important because it:

- runs commands in sequence,
- resets closed-window cache for mutating commands,
- calls `refreshModel_nonCancellable()` after commands,
- preserves command stdout/stderr/exit status behavior.

### Adding a command checklist

1. Add `Sources/Common/cmdArgs/impl/<Name>CmdArgs.swift`.
2. Add case to `CmdKind` and parser mapping in `Sources/Common/cmdArgs/cmdArgsManifest.swift`.
3. Add command implementation in `Sources/AppBundle/command/impl/<Name>Command.swift`.
4. Add mapping in `Sources/AppBundle/command/cmdManifest.swift`.
5. Add docs file `docs/aerospace-<name>.adoc` and include it in `docs/commands.adoc`.
6. Update `grammar/commands-bnf-grammar.txt` if shell completion grammar should know it.
7. Regenerate help with `./script/generate-cmd-help.sh` or `./generate.sh`.
8. Add parser/execution tests in `Sources/AppBundleTests/command/`.

## 6. Core tree model

AeroSpace maintains an in-memory tree of workspaces, containers, and windows. Most model mutation is `@MainActor`.

Key files:

| File | Purpose |
| --- | --- |
| `tree/TreeNode.swift` | Base class: children, parent, MRU, binding, weights, layout rect caches. |
| `tree/TreeNodeCases.swift` | Exhaustive taxonomy and legal parent-child relations. |
| `tree/TreeNodeEx.swift` | Tree traversal helpers: workspace/monitor lookup, leaf lookup, MRU window lookup. |
| `tree/Workspace.swift` | Workspace registry, monitor visibility maps, monitor rearrangement. |
| `tree/WorkspaceEx.swift` | Lazy workspace child containers and force assignment resolution. |
| `tree/Window.swift` | Abstract window leaf API. |
| `tree/MacWindow.swift` | Concrete AX-backed window node. |
| `tree/MacApp.swift` | Per-application AX state/thread and native operations. |
| `tree/FloatingWindowsContainer.swift` | Floating-window shim container. |
| `tree/MacosUnconventionalWindowsContainer.swift` | Minimized/fullscreen/hidden/popup state containers. |

### Legal topology

The legal topology is enforced by `getChildParentRelation` in `TreeNodeCases.swift`.

Typical workspace subtree:

```text
Workspace
  ├─ TilingContainer              # root tiling container
  │   ├─ Window
  │   └─ TilingContainer
  │       └─ Window
  ├─ FloatingWindowsContainer
  │   └─ Window
  ├─ MacosFullscreenWindowsContainer
  └─ MacosHiddenAppsWindowsContainer

Global-ish unconventional containers:
  ├─ macosMinimizedWindowsContainer
  └─ macosPopupWindowsContainer
```

Important invariants:

- Windows should not be direct children of `Workspace`; floating windows live in `FloatingWindowsContainer`.
- A workspace has at most one root tiling container and at most one floating container. These are lazily created.
- `TreeNode.bind` is the central mutation primitive. It validates relations and updates MRU.
- `unbindFromParent` must be used when moving nodes; direct mutation of children is unsafe.
- `Window.get(byId:)` uses `MacWindow.allWindowsMap` in production and test workspaces in unit tests.

### MRU behavior

Each `TreeNode` has an MRU stack of children. Binding or explicit focus can call `markAsMostRecentChild()`, which propagates up the tree.

MRU is used by:

- workspace focus fallback (`Workspace.toLiveFocus()`),
- tiling insertion next to most recent window,
- focus commands,
- cleanup after closed windows.

Be careful when binding windows as a side effect of detection or normalization: binding updates MRU and can alter future focus decisions.

## 7. Workspace and monitor model

Monitors are abstracted by `Monitor` in `model/Monitor.swift`. Real monitors are `LazyMonitor` wrappers around `NSScreen`; tests can mock monitors.

A workspace's monitor is not simply its tree parent. It is resolved by:

```swift
forceAssignedMonitor
    ?? visibleWorkspaceToScreenPoint[self]?.monitorApproximation
    ?? assignedMonitorPoint?.monitorApproximation
    ?? mainMonitor
```

Important maps in `Workspace.swift`:

- `screenPointToVisibleWorkspace: [CGPoint: Workspace]`
- `visibleWorkspaceToScreenPoint: [Workspace: CGPoint]`
- `screenPointToPrevVisibleWorkspace: [CGPoint: String]`
- `assignedMonitorPoint` on each workspace

The active workspace of a monitor is keyed by monitor top-left point.

### Monitor rearrangement

`rearrangeWorkspacesOnMonitors()` handles display topology changes:

1. Compute new monitor top-left points.
2. Map old visible workspaces to nearest new monitors.
3. Apply one-shot on-connect assignments by updating `assignedMonitorPoint` and moving visible configured workspaces if needed.
4. Fill any unoccupied monitor with a stub workspace.

This function is triggered from `Monitor.activeWorkspace` when mappings are stale and from `gcMonitors()` during heavy refresh.

### Moving workspaces

`move-workspace-to-monitor` calls `Monitor.setActiveWorkspace`. This is blocked only by permanent force assignment (`workspace-to-monitor-force-assignment`), via `isValidAssignment`.

One-shot connect assignment intentionally does not block later moves.

## 8. Window/app and macOS Accessibility integration

AeroSpace interacts with macOS windows through Accessibility (AX) and a small amount of private API.

Key files:

| File | Purpose |
| --- | --- |
| `tree/MacApp.swift` | Owns AX app object, per-app thread/CFRunLoop, AX windows map, focus/frame/native state operations. |
| `tree/MacWindow.swift` | Concrete window registration and window-level AX adapter. |
| `util/accessibility.swift` | AX permission loop and AX attribute wrappers. |
| `util/AxSubscription.swift` | AXObserver subscription lifecycle. |
| `util/AxUiElementMock.swift` | Protocol used for real AX elements and test mocks. |
| `windowLevelCache.swift` | CG window-level cache via `CGWindowListCopyWindowInfo`. |
| `Sources/PrivateApi/include/private.h/.m` | Private C bridge. |

### Per-app AX threading model

Each `MacApp` creates a dedicated `Thread` with a CFRunLoop. AX objects are guarded by `ThreadGuardedValue`, and work is submitted via `runInLoop`/`runInLoopAsync`.

This matters because AX calls can block or be thread-sensitive. Do not access `AXUIElement` directly from arbitrary threads.

### Window discovery

Heavy refresh performs discovery:

```text
MacApp.refreshAllAndGetAliveWindowIds
  -> iterate running apps
  -> register MacApp per NSRunningApplication
  -> read AX windows
  -> return [MacApp: [windowId]]

refresh()
  -> garbage collect disappeared MacWindow objects
  -> MacWindow.getOrRegister for new IDs
```

`MacWindow.getOrRegister`:

1. Reads initial AX rect.
2. Chooses initial workspace:
   - startup: monitor under rect center,
   - non-startup: current focused workspace.
3. Classifies AX window type as popup/dialog/window.
4. Binds to popup container, floating container, or tiling tree.
5. Registers in `MacWindow.allWindowsMap`.
6. Runs closed-window cache restore or `on-window-detected` callbacks.

### Native focus

`MacApp.nativeFocus` pushes AeroSpace focus back to macOS. It uses an AltTab-style sequence:

1. Deminiaturize target window if needed.
2. Set AX main and raise target before fronting the app.
3. Use private SkyLight focus helper for the specific window.
4. Raise target again.

The private helper in `PrivateApi` dynamically resolves SkyLight symbols with `dlopen`/`dlsym` to avoid link-time private-framework failures.

### Private APIs

Private APIs are isolated to `Sources/PrivateApi`.

Currently important helpers:

- `_AXUIElementGetWindow`: map AX window element to `CGWindowID`.
- `ASAltTabFocusWindow`: robust process/window focus using SkyLight `_SLPSSetFrontProcessWithOptions` plus synthetic WindowServer make-key events.

When adding private API usage, prefer wrapping it in `PrivateApi` rather than scattering `@_silgen_name` or C declarations in Swift code.

## 9. Focus model

Focus is represented separately from macOS native focus.

Key files:

- `focus.swift`
- `focusCache.swift`
- native focus push in `MacApp.nativeFocus`

Types:

- `LiveFocus`: current resolved focus object (`windowOrNil`, `workspace`).
- `FrozenFocus`: stable by-value snapshot (`windowId`, `workspaceName`, monitor id).
- global `@MainActor var focus: LiveFocus`.

Focus flow:

```text
native macOS focus changes
  -> getNativeFocusedWindow
  -> updateFocusCache(nativeFocused)
  -> nativeFocused.focusWindow()
  -> setFocus(to:)

AeroSpace command changes focus
  -> setFocus(to:)
  -> light session notices focusBefore != focusAfter
  -> focusAfter?.nativeFocus()
```

`setFocus(to:)` also:

- updates active workspace on the workspace monitor,
- records previous workspace/focus for back-and-forth commands,
- marks focused window as MRU.

Focus callbacks are emitted from `checkOnFocusChangedCallbacks_nonCancellable`.

## 10. Refresh sessions and event lifecycle

Most external changes enter one of two session types in `layout/refresh.swift`.

### Heavy refresh

Used for full reconciliation after notifications.

```text
runHeavyCompleteRefreshSession(event)
  -> read native focused window
  -> updateFocusCache
  -> optional pre-layout
  -> refreshModel_nonCancellable
  -> refresh()                   # app/window discovery + GC
  -> gcMonitors()
  -> update tray / panels
  -> normalizeLayoutReason()
  -> layoutWorkspaces()
```

Scheduled by:

- AX notifications (`refreshObs`),
- NSWorkspace notifications,
- active Space changes,
- config watcher,
- post-command follow-up.

### Light session

Used around commands/mouse interactions.

```text
runLightSession(event, guard) {
  -> cancel active heavy refresh
  -> read native focus
  -> updateFocusCache
  -> refreshModel before body
  -> run body
  -> refreshModel after body
  -> layoutWorkspaces
  -> if focus changed: nativeFocus
  -> schedule follow-up heavy refresh
}
```

Light sessions keep interactions responsive and deterministic while still allowing a full reconciliation afterward.

### RefreshSessionEvent

Defined in `Sources/Common/util/commonUtil.swift`. Important flags:

- `isStartup`: used by startup-only behavior.
- `isFocusFollowsMouse`: avoids some expensive/recursive behavior during focus-follows-mouse.

## 11. Layout subsystem

Layout is in `Sources/AppBundle/layout/`.

Important files:

- `layoutRecursive.swift`: recursive geometry calculation and AX frame application.
- `refresh.swift`: orchestration and `layoutWorkspaces()` scheduling.

Layout flow:

```text
layoutWorkspaces
  -> visible workspaces: unhide + layoutWorkspace
  -> invisible workspaces: hide windows in monitor corner

Workspace.layoutWorkspace
  -> resolved monitor gaps
  -> root tiling container layoutRecursive
  -> floating container layoutRecursive
  -> fullscreen container layoutRecursive
```

`layoutRecursive` writes:

- `lastAppliedLayoutVirtualRect`
- `lastAppliedLayoutPhysicalRect`

These caches are used by mouse movement/resizing and focus-follows-mouse. Preserve them when changing layout behavior.

Tiling containers:

- normalize child weights,
- split the available rect by orientation,
- account for gaps,
- recurse into children.

Floating windows:

- have remembered physical rects,
- can be moved across monitors/workspaces while preserving relative position,
- are handled through `FloatingWindowsContainer` rather than direct workspace children.

## 12. macOS native/unconventional window states

AeroSpace models macOS states explicitly:

- native fullscreen,
- minimized,
- hidden app windows,
- popups/dialogs.

`normalizeLayoutReason.swift` moves windows between standard layout and unconventional containers based on AX state.

`LayoutReason` on `Window` records whether the current tree placement is standard or macOS-driven. When a window exits fullscreen/minimized/hidden state, `exitMacOsNativeUnconventionalState` restores it to an appropriate parent.

Important behavior:

- Popups are usually ignored for normal focus/layout.
- Minimized windows are stored in `macosMinimizedWindowsContainer`.
- Hidden/fullscreen windows are workspace-specific unconventional containers.
- Cross-workspace floating behavior has special normalization and cache interactions.

## 13. Closed windows cache

File: `tree/frozen/closedWindowsCache.swift`.

Purpose: protect against macOS/AX glitches, especially lock screen, where all windows may appear to disappear temporarily. AeroSpace caches the world when windows close so it can restore the tree when they reappear.

Main functions:

- `cacheClosedWindowIfNeeded()`
- `restoreClosedWindowsCacheIfNeeded(newlyDetectedWindow:)`
- `resetClosedWindowsCache()`

Be careful: this cache can also restore old workspace state for apps that reuse window IDs. Recent cross-workspace floating fixes intentionally avoid caching/restoring some floating windows when `cross-workspace-floating-windows` is enabled.

Commands declare whether they reset the cache through `shouldResetClosedWindowsCache`.

## 14. Observer/event sources

Key event sources:

| Source | File | Effect |
| --- | --- | --- |
| AX app/window notifications | `MacApp.swift`, `AxSubscription.swift` | Schedule heavy refresh, moved/resize handling. |
| NSWorkspace notifications | `GlobalObserver.swift` | App activation/launch/hide/unhide/Space changes. |
| Global mouse-up monitor | `GlobalObserver.swift` | Detect monitor desktop clicks and trigger refresh. |

Start with:

- `Workspace.workspaceMonitor`

- `rearrangeWorkspacesOnMonitors`

Watch for:

- forced assignment vs one-shot assignment,
- visible mapping vs assigned monitor point,
- stub workspace generation,

Start with:

- `layoutRecursive.swift`

- `Common/cmdArgs/impl/*CmdArgs.swift`

## 18. Invariants and pitfalls

- Model mutation should be on the main actor.
- AX calls should go through `MacApp` thread helpers.
- Do not directly edit `TreeNode.children`; use `bind`/`unbindFromParent`.
- Window tree placement and macOS native focus are separate systems; syncing them too eagerly can cause loops.
- Workspace visibility is a monitor-point mapping, not a tree edge.
- Empty workspaces may be garbage-collected unless persistent/configured/visible/focused.
- Binding updates MRU. This is useful but can surprise focus behavior.
- `workspace-to-monitor-force-assignment` is permanent and blocks moves; `workspace-to-monitor-assignment-on-connect` only sets default placement on startup/connect.
- `closedWindowsCache` is important for lock-screen/AX glitches but can restore stale workspace state if overused.
- Generated files should not be edited manually unless that is the established source of truth.
